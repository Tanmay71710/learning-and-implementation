1. Overview

The Script Execution Manager project now supports multiple environments to enable proper development workflow, testing, and production deployment. This multi-environment setup provides isolation, scalability, and environment-specific configurations for different stages of the development lifecycle.

Key Benefits:
  • Environment isolation for development, testing, and production
  • Resource optimization with environment-specific configurations
  • Automated deployment pipelines based on git branches
  • GitOps integration with ArgoCD
  • Flexible deployment methods (Kustomize, Helm, ArgoCD)

Supported Environments

  • dev Development - Development environment with debug enabled
  • staging Staging - Pre-production testing environment
  • pub/dev Public Dev - Public development environment for external testing
  • production Production - Production environment for live traffic

3.2 Deployment Methods

Kustomize Deployment

# Deploy to specific environment kubectl apply -k environments/dev/k8s kubectl apply -k environments/staging/k8s kubectl apply -k environments/pub-dev/k8s kubectl apply -k environments/production/k8s

Helm Deployment

# Deploy to specific environment helm install script-execution-manager ./helm/script-execution-manager \ -f environments/dev/helm-values.yaml \ --namespace script-execution-manager-dev --create-namespace

ArgoCD Deployment

# Apply environment-specific ArgoCD application kubectl apply -f environments/dev/argocd-application.yaml # Or use app-of-apps for all environments kubectl apply -f argocd/app-of-apps.yaml

3.3 GitHub Actions CI/CD

The GitHub Actions workflow automatically deploys to environments based on the branch that receives changes:

  • Push to develop branch → dev environment deployment
  • Push to staging branch → staging environment deployment
  • Push to pub/dev branch → pub-dev environment deployment
  • Push to master branch → production environment deployment
Manual Deployment: You can also trigger manual deployments using the workflow dispatch feature in GitHub Actions, selecting the specific target environment.

4. Deployment Scenarios

Scenario 1: New Feature Development

Description: A developer wants to implement a new feature and test it in the development environment before promoting to other environments.

Steps:

  1. Create a feature branch from develop: git checkout -b feature/new-feature develop
  2. Implement the feature and commit changes
  3. Push to feature branch: git push origin feature/new-feature
  4. Create pull request to develop branch
  5. Get team approval and ensure CI/CD checks pass
  6. Merge to develop branch
  7. GitHub Actions automatically deploys to dev environment
  8. Test the feature in dev environment

Expected Outcome: Feature is deployed to dev environment and available for testing.

Scenario 2: Promoting Feature to Staging

Description: A feature has been tested in dev and is ready for staging environment testing.

Steps:

  1. Ensure feature is working correctly in dev environment
  2. Checkout staging branch: git checkout staging
  3. Merge develop into staging: git merge develop
  4. Push to staging: git push origin staging
  5. GitHub Actions automatically deploys to staging environment
  6. Perform comprehensive testing in staging environment

Expected Outcome: Feature is deployed to staging environment for pre-production testing.

Scenario 3: Public Development Testing

Description: A feature needs external testing by stakeholders or beta testers in the pub-dev environment.

Steps:

  1. Ensure feature is tested in staging environment
  2. Checkout pub/dev branch: git checkout pub/dev
  3. Merge staging into pub/dev: git merge staging
  4. Push to pub/dev: git push origin pub/dev
  5. GitHub Actions automatically deploys to pub-dev environment
  6. Share external access URL with stakeholders
  7. Collect feedback and perform testing

Expected Outcome: Feature is deployed to pub-dev environment and accessible for external testing.

Scenario 4: Production Deployment

Description: A feature has completed all testing phases and is ready for production deployment.

Steps:

  1. Ensure feature is tested in staging and pub-dev environments
  2. Get final approval from stakeholders
  3. Create release tag: git tag -a v1.0.0 -m "Release version 1.0.0"
  4. Checkout master branch: git checkout master
  5. Merge staging into master: git merge staging
  6. Push to master: git push origin master
  7. Push tag: git push origin v1.0.0
  8. GitHub Actions automatically deploys to production environment
  9. Verify production deployment
  10. Monitor production metrics and logs

Expected Outcome: Feature is deployed to production environment and live for users.

Scenario 5: Emergency Hotfix

Description: A critical bug is discovered in production that needs immediate fixing.

Steps:

  1. Identify the critical issue and impact
  2. Checkout master branch: git checkout master
  3. Create hotfix branch: git checkout -b hotfix/critical-bug-fix
  4. Implement the emergency fix
  5. Test the fix locally
  6. Commit and push hotfix branch
  7. Create pull request to master
  8. Get expedited approval and merge to master
  9. GitHub Actions immediately deploys to production
  10. Verify fix in production environment
  11. Back-merge hotfix to develop branch: git checkout develop && git merge hotfix/critical-bug-fix
  12. Delete hotfix branch

Expected Outcome: Critical bug is fixed and deployed to production environment quickly, with the fix also available in development for future releases.

Scenario 6: Manual Deployment to Specific Environment

Description: Need to deploy to a specific environment without changing branches or for testing purposes.

Steps:

  1. Go to repository's Actions tab in GitHub
  2. Select "Build and Push Docker Image to Artifactory" workflow
  3. Click "Run workflow" button
  4. Select target environment from dropdown (dev, staging, pub-dev, production)
  5. Optionally specify custom image tag
  6. Enable/disable Kubernetes deployment as needed
  7. Click "Run workflow" to start the process
  8. Monitor workflow execution in Actions tab

Expected Outcome: Docker image is built and deployed to the specified environment without requiring branch changes.

Scenario 7: Rollback Deployment

Description: A deployment causes issues and needs to be rolled back to the previous version.

Steps:

  1. Identify the problematic deployment
  2. Check ArgoCD application history for previous versions
  3. In ArgoCD UI, navigate to the application
  4. Click "History and Rollback"
  5. Select the previous stable version
  6. Click "Rollback" to revert to that version
  7. Verify the rollback was successful
  8. Monitor application health
  9. Investigate the issue with the problematic version

Expected Outcome: Application is rolled back to the previous stable version, minimizing downtime and impact.

Complete Feature Development Workflow

Description: End-to-end workflow from feature conception to production deployment.

Phase 1: Development

  1. Create feature branch from develop
  2. Implement feature with proper testing
  3. Create pull request to develop
  4. Get code review approval
  5. Merge to develop (auto-deploys to dev)
  6. Test in dev environment

Phase 2: Staging

  1. Promote feature to staging branch
  2. Deploy to staging environment
  3. Perform comprehensive testing
  4. Get QA approval

Phase 3: Public Testing

  1. Promote to pub-dev branch (if external testing needed)
  2. Deploy to pub-dev environment
  3. Conduct beta testing
  4. Collect feedback and make adjustments

Phase 4: Production

  1. Final approval and release planning
  2. Create release tag
  3. Merge to master branch
  4. Deploy to production environment
  5. Monitor production metrics
  6. Post-deployment verification

5.2 Git Workflow Scenarios

Scenario: Parallel Feature Development

Description: Multiple developers working on different features simultaneously.

Workflow:

  1. Developer A creates feature/auth from develop
  2. Developer B creates feature/dashboard from develop
  3. Both developers work independently
  4. Developer A completes and merges to develop
  5. Developer B pulls latest develop changes
  6. Developer B resolves any conflicts
  7. Developer B completes and merges to develop
  8. Both features are available in dev environment

Expected Outcome: Multiple features developed in parallel with proper conflict resolution.

Scenario: Release Management

Description: Managing releases and versioning across environments.

Workflow:

  1. Plan release with feature freeze date
  2. Complete all feature development
  3. Perform testing in staging environment
  4. Create release branch: git checkout -b release/v1.0.0 staging
  5. Perform final testing and bug fixes
  6. Merge release to master
  7. Create release tag: git tag -a v1.0.0
  8. Deploy to production
  9. Merge release back to develop
  10. Delete release branch

Expected Outcome: Structured release process with proper versioning and testing.

Scenario: Environment-Specific Configuration

Description: Managing different configurations for different environments.

Workflow:

  1. Update base configuration in k8s-base/
  2. Modify environment-specific overlays as needed
  3. Update environment .env files with appropriate values
  4. Update Helm values files for environment-specific settings
  5. Test changes in dev environment first
  6. Promote configuration changes through environments
  7. Document any configuration differences

Expected Outcome: Proper configuration management across all environments.

6. Troubleshooting Scenarios

Scenario 1: Deployment Failure in Specific Environment

Description: Deployment fails in staging environment but works in dev.

Investigation Steps:

  1. Check GitHub Actions workflow logs for staging deployment
  2. Compare environment configurations between dev and staging
  3. Check resource limits and requests in staging
  4. Verify database connection strings for staging
  5. Check secrets and credentials for staging environment
  6. Review Kubernetes events in staging namespace
  7. Examine pod logs in staging environment

Common Solutions:

  • Adjust resource limits for staging environment
  • Fix environment-specific configuration issues
  • Update secrets and credentials
  • Resolve database connectivity issues

Scenario 2: Wrong Environment Deployed

Description: Code was deployed to the wrong environment.

Investigation Steps:

  1. Check which branch was pushed
  2. Verify GitHub Actions environment detection logic
  3. Review ArgoCD application target branches
  4. Check if manual deployment was triggered incorrectly

Solutions:

  • Roll back the incorrect deployment
  • Deploy to the correct environment
  • Review and fix branch naming conventions
  • Update workflow environment detection if needed

Scenario 3: Resource Constraints in Production

Description: Production environment is experiencing resource constraints.

Investigation Steps:

  1. Monitor resource usage in production namespace
  2. Check pod resource requests vs limits
  3. Review autoscaling configuration
  4. Analyze application performance metrics
  5. Check cluster capacity

Solutions:

  • Adjust resource limits in production configuration
  • Modify autoscaling parameters
  • Optimize application performance
  • Request additional cluster capacity
  • Implement resource quotas if needed

Scenario 4: Database Connection Issues

Description: Application cannot connect to database in specific environment.

Investigation Steps:

  1. Check database connection string in environment .env file
  2. Verify database is running and accessible
  3. Check network policies and security groups
  4. Review database credentials and secrets
  5. Test database connectivity from pod

Solutions:

  • Update connection strings with correct values
  • Fix database credentials in secrets
  • Configure network policies to allow database access
  • Ensure database is running and healthy
  • Update DNS configuration if needed

Scenario 5: ArgoCD Sync Failures

Description: ArgoCD applications are failing to sync with git repository.

Investigation Steps:

  1. Check ArgoCD application sync status
  2. Review sync operation logs in ArgoCD
  3. Verify git repository accessibility
  4. Check branch and path configurations
  5. Review Kubernetes cluster connectivity
  6. Check for resource conflicts

Solutions:

  • Fix git repository connection issues
  • Update branch references if changed
  • Resolve resource conflicts in cluster
  • Check ArgoCD server health
  • Verify RBAC permissions

Scenario 6: Git Merge Conflicts

Description: Merge conflicts occur when promoting code between environments.

Investigation Steps:

  1. Identify conflicting files
  2. Review changes in both branches
  3. Understand the context of conflicts
  4. Communicate with team if needed

Solutions:

  • Resolve conflicts manually
  • Use git merge tools for assistance
  • Consult with team for conflicting changes
  • Test resolution thoroughly
  • Document conflict resolution patterns

Scenario 7: CI/CD Pipeline Failures

Description: GitHub Actions workflow is failing for specific environments.

Investigation Steps:

  1. Check workflow logs in GitHub Actions
  2. Identify which step is failing
  3. Review error messages and stack traces
  4. Check for environment-specific issues
  5. Verify secrets and credentials
  6. Review workflow configuration

Solutions:

  • Fix code issues causing test failures
  • Update workflow configuration
  • Add missing secrets and credentials
  • Fix Docker build issues
  • Resolve Kubernetes deployment problems

7. Best Practices

7.1 Development Practices

  • Always test in dev first before promoting to other environments
  • Use feature branches for development work
  • Create pull requests for all merges
  • Get code reviews before merging
  • Write tests for new features
  • Update documentation with changes
  • Follow commit message conventions

7.2 Deployment Practices

  • Follow environment promotion order: dev → staging → pub/dev → production
  • Test thoroughly in each environment before promotion
  • Use semantic versioning for releases
  • Create release tags on master branch
  • Monitor deployments after each promotion
  • Have rollback plans ready for production deployments
  • Use hotfix branches for production emergencies

7.3 Configuration Management

  • Never commit secrets to the repository
  • Use environment variables for sensitive data
  • Maintain separate databases for each environment
  • Document configuration differences between environments
  • Use Kustomize overlays for environment-specific changes
  • Regular security audits for all environments
  • Rotate credentials regularly

7.4 Monitoring and Logging

  • Monitor all environments continuously
  • Set up alerts for critical metrics
  • Review logs regularly across all environments
  • Use centralized logging for better analysis
  • Track deployment metrics and success rates
  • Monitor resource usage and optimize as needed
  • Set up health checks for all applications

7.5 Security Practices

  • Implement network policies for environment isolation
  • Use RBAC for access control
  • Enable branch protection for main branches
  • Require approvals for production deployments
  • Use secrets management for sensitive data
  • Regular security scanning of containers and code
  • Implement audit logging for sensitive operations

7.6 Git Workflow Best Practices

  • Keep branches up to date before merging
  • Use descriptive branch names
  • Delete merged branches to keep repository clean
  • Use squash merge for feature branches
  • Use merge commits for environment promotions
  • Never force push to shared branches
  • Resolve conflicts locally before pushing

7.7 Emergency Procedures

  • Have rollback procedures documented and tested
  • Establish communication channels for incidents
  • Create on-call rotation for production issues
  • Document incident response procedures
  • Practice incident drills regularly
  • Maintain service status page for stakeholders
  • Post-incident analysis for continuous improvement
