A. Title and Overview

Kubernetes Container Orchestration

Role in Project: Kubernetes serves as the deployment platform for the Script Execution Manager, providing container orchestration, scaling, and management for the Flask application and supporting services.

Why Kubernetes is Used

  • Container Orchestration: Automated deployment, scaling, and management of containerized applications
  • Self-Healing: Automatic restart of failed containers and replacement of unhealthy nodes
  • Scalability: Horizontal scaling with Horizontal Pod Autoscaler (HPA)
  • Service Discovery: Built-in service discovery and load balancing
  • Declarative Configuration: Infrastructure as Code with YAML manifests

GitOps Pipeline Position

Kubernetes is the final destination in the GitOps pipeline. Argo CD monitors the Git repository and automatically applies Kubernetes manifests to deploy and manage the application in the cluster.

Key Responsibilities

  • Container orchestration and scheduling
  • Service discovery and load balancing
  • Configuration management with ConfigMaps and Secrets
  • Storage orchestration with PersistentVolumes
  • Self-healing and automatic scaling
  • Rolling updates and rollbacks

B. What You Should Know Before Starting

Foundational Concepts

  • Container Concepts: Understanding of Docker containers and images

D. Installation / Setup

Step 1: Verify Cluster Access

# Verify cluster connectivity
kubectl cluster-info

# Check cluster nodes
kubectl get nodes

# List namespaces
kubectl get namespaces

Step 2: Create Namespace

# Create namespace for application
kubectl create namespace script-execution-manager

# Verify namespace creation
kubectl get namespace script-execution-manager

Step 3: Create ConfigMap

# Apply ConfigMap from k8s directory
kubectl apply -f k8s/configmap.yaml -n script-execution-manager

# Verify ConfigMap
kubectl get configmap -n script-execution-manager

Step 4: Create Secrets

# Apply Secrets from k8s directory
kubectl apply -f k8s/secrets.yaml -n script-execution-manager

# Verify Secrets
kubectl get secrets -n script-execution-manager

Step 5: Create Docker Registry Secret

# Create Docker registry secret for Artifactory
kubectl create secret docker-registry artifactory-registry-secret \
  --docker-server=artifactory.local:8082 \
  --docker-username=admin \
  --docker-password=password \
  -n script-execution-manager

Step 6: Deploy Application

# Apply Deployment
kubectl apply -f k8s/deployment.yaml -n script-execution-manager

# Apply Service
kubectl apply -f k8s/service.yaml -n script-execution-manager

Step 7: Validate Deployment

# Check pods
kubectl get pods -n script-execution-manager

# Check services
kubectl get svc -n script-execution-manager

# Check deployment status
kubectl get deployment -n script-execution-manager

# View pod logs
kubectl logs -n script-execution-manager deployment/script-execution-manager
Tip: Use kubectl's --watch flag to monitor pod status in real-time: kubectl get pods -n script-execution-manager --watch

Common Configuration Mistakes

Warning: Avoid using imagePullPolicy: Always in production without proper image caching, as it can cause unnecessary network traffic and slow deployments.
  • Setting incorrect resource limits causing pod eviction
  • Missing health checks causing unhealthy pods to remain in service
  • Incorrect selector causing service to not find pods
  • Using wrong port numbers in service configuration
  • Not setting appropriate restart policies

F. Cookbook Recipes

Recipe 1: Basic Application Deployment

Purpose: Deploy a basic Flask application to Kubernetes

Ingredients: Docker image, Kubernetes manifests, kubectl access

Steps:

  1. Build and push Docker image to registry
  2. Create namespace: kubectl create namespace app
  3. Create ConfigMap with environment variables
  4. Create Secret with sensitive data
  5. Apply Deployment manifest
  6. Apply Service manifest
  7. Verify pod status and connectivity

Expected Result: Application running with 3 replicas and load-balanced service

Verification: kubectl get pods -n app and test service access

Recipe 2: Configure Horizontal Pod Autoscaler

Purpose: Enable automatic scaling based on CPU/memory usage

Ingredients: Metrics Server, HPA configuration

Steps:

  1. Ensure Metrics Server is installed in cluster
  2. Set resource requests on Deployment (required for HPA)
  3. Create HPA YAML manifest
  4. Configure target CPU utilization (e.g., 70%)
  5. Set min/max replica limits
  6. Apply HPA: kubectl apply -f hpa.yaml
  7. Test scaling by increasing load

Expected Result: Pods automatically scale based on CPU utilization

Verification: kubectl get hpa and monitor replica changes

Recipe 3: Set up Ingress for External Access

Purpose: Configure external access via Ingress controller

Ingredients: Ingress controller, Ingress resource, DNS configuration

Steps:

  1. Ensure Ingress controller is running (Traefik, NGINX)
  2. Create Ingress resource YAML
  3. Configure host rules and paths
  4. Set up TLS certificate if using HTTPS
  5. Configure DNS to point to Ingress IP
  6. Apply Ingress: kubectl apply -f ingress.yaml
  7. Test external access via configured hostname

Expected Result: Application accessible via external hostname

Verification: Test access from external network

Recipe 4: Configure Persistent Storage

Purpose: Add persistent storage for application data

Ingredients: StorageClass, PersistentVolumeClaim, Pod configuration

Steps:

  1. Check available StorageClasses in cluster
  2. Create PersistentVolumeClaim YAML
  3. Set storage size and access mode
  4. Configure volumeMounts in Deployment
  5. Apply PVC: kubectl apply -f pvc.yaml
  6. Apply updated Deployment with volume mounts
  7. Verify storage is mounted and accessible

Expected Result: Application has persistent storage that survives pod restarts

Verification: kubectl exec -it pod -- df -h to check mounted storage

Recipe 5: Perform Rolling Update

Purpose: Update application with zero downtime

Ingredients: New Docker image, Deployment manifest

Steps:

  1. Build and push new Docker image version
  2. Update image tag in Deployment manifest
  3. Apply updated manifest: kubectl apply -f deployment.yaml
  4. Monitor rollout status: kubectl rollout status deployment/app
  5. Verify new pods are running and healthy
  6. Test application functionality

Expected Result: Application updated with zero downtime using rolling strategy

Verification: kubectl get pods shows new pod versions

H. Best Practices

Naming Conventions

  • Use lowercase with hyphens for resource names: script-execution-manager
  • Use descriptive labels: app: script-execution-manager, version: v1.0.0
  • Use semantic versioning for image tags: v1.0.0, v1.1.0
  • Use consistent naming across environments: script-execution-manager-dev, script-execution-manager-prod

Resource Management

  • Always set resource requests and limits for pods
  • Use appropriate resource limits based on application requirements
  • Configure Horizontal Pod Autoscaler for production workloads
  • Implement resource quotas at namespace level
  • Monitor resource usage and optimize accordingly

Configuration Management

  • Use ConfigMaps for non-sensitive configuration data
  • Use Secrets for sensitive data (credentials, API keys)
  • Never commit Secrets to version control
  • Use environment-specific ConfigMaps and Secrets
  • Implement configuration versioning with Git

Security Hardening

  • Implement Network Policies to restrict pod communication
  • Use RBAC for access control and permissions
  • Run containers as non-root users when possible
  • Implement pod security policies
  • Regular security updates and patching

Monitoring and Logging

  • Implement comprehensive logging with centralized aggregation
  • Set up health checks and liveness probes
  • Monitor resource usage and set up alerts
  • Implement distributed tracing for microservices
  • Set up dashboards for cluster and application monitoring

Deployment Strategies

  • Use rolling updates for zero-downtime deployments
  • Implement proper rollback procedures
  • Use canary deployments for major version changes
  • Implement blue-green deployments for critical applications
  • Test deployments in staging before production

J. Troubleshooting

Problem: Pods stuck in ImagePullBackOff state ▼

Symptoms: Pods show ImagePullBackOff status, container never starts

Likely Cause: Invalid image reference, missing imagePullSecrets, registry authentication issues

How to Diagnose: kubectl describe pod, check image reference, verify registry access

Fix: Correct image reference, create imagePullSecrets, update registry credentials

Prevention: Validate image references, test registry access, configure proper secrets

Problem: Pods stuck in CrashLoopBackOff state ▼

Symptoms: Pods repeatedly start and crash, never become healthy

Likely Cause: Application errors, missing dependencies, incorrect configuration

How to Diagnose: Check pod logs, examine application errors, verify configuration

Fix: Fix application errors, add missing dependencies, correct configuration

Prevention: Implement proper startup validation, use health checks, test locally first

Problem: Service cannot reach pods ▼

Symptoms: Service shows no endpoints, traffic not reaching pods

Likely Cause: Incorrect selector, pod labels mismatch, network policies blocking traffic

How to Diagnose: Check service endpoints, verify pod labels, examine network policies

Fix: Correct service selector, update pod labels, adjust network policies

Prevention: Validate label matching, test connectivity, review network policies

Problem: High memory usage causing pod eviction ▼

Symptoms: Pods evicted due to memory pressure, application instability

Likely Cause: Memory leaks, insufficient resource limits, over-provisioning

How to Diagnose: Monitor memory usage, check resource limits, analyze application memory patterns

Fix: Fix memory leaks, increase resource limits, optimize application memory usage

Prevention: Set appropriate resource limits, monitor usage, implement memory profiling

K. Q&A / FAQ

Q: Why is Kubernetes used instead of traditional deployment methods?
A: Kubernetes provides automated scaling, self-healing, and declarative configuration management that traditional methods cannot match. It enables GitOps workflows and provides enterprise-grade reliability.
Q: What happens if Kubernetes configuration is wrong?
A: Deployments may fail, pods may not start, or applications may be inaccessible. Always validate configuration with dry-run and test in non-production environments first.
Q: How do I validate Kubernetes deployment configuration?
A: Use kubectl apply --dry-run to test configuration, kubectl explain to check field documentation, and kubectl validate for schema validation.
Q: How do I debug Kubernetes deployment failures?
A> Check pod status with kubectl get pods, view logs with kubectl logs, describe pods with kubectl describe pod, and check events with kubectl get events.
Q: How do I rollback Kubernetes deployment changes?
A: Use kubectl rollout undo deployment/app to rollback to previous version, or kubectl rollout undo deployment/app --to-revision=2 for specific revision.
Q: What logs should I inspect for troubleshooting?
A: Pod logs, kubelet logs, Kubernetes control plane logs, application logs, and any sidecar container logs. Use kubectl logs and cluster logging systems.
Q: How do I secure Kubernetes credentials?
A: Use RBAC for access control, rotate certificates regularly, use service accounts for automation, and integrate with secret management systems like HashiCorp Vault.
Q: How does Kubernetes behave in production?
A: In production, Kubernetes handles high-traffic loads, provides self-healing, enables horizontal scaling, and integrates with monitoring and logging systems for enterprise-grade reliability.
Q: How do I improve Kubernetes performance?
A: Optimize resource requests/limits, use horizontal pod autoscaling, implement cluster autoscaling, use node affinity rules, and tune kubelet and cluster settings.
Q: What are the common limitations of Kubernetes?
A: Complexity of setup and management, resource overhead, learning curve, networking complexity, and potential cost for cloud-hosted clusters.
