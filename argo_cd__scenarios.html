1. Basic Application Deployment

Use Case

Overview: Deploy a simple nginx application using ArgoCD with auto-sync enabled. This is the foundational scenario for understanding how ArgoCD manages Kubernetes deployments using GitOps methodology.

What This Use Case Covers:

  • Setting up your first ArgoCD Application manifest
  • Connecting ArgoCD to a Git repository containing Kubernetes manifests
  • Configuring automated synchronization between Git and cluster
  • Understanding the basic deployment workflow in ArgoCD
  • Verifying successful deployment and health status

When to Use This Scenario:

  • You're new to ArgoCD and want to understand the basic deployment workflow
  • You need to deploy a simple stateless application like a web server
  • You're setting up a proof-of-concept for GitOps in your organization
  • You want to understand how auto-sync works before applying it to complex applications
  • You're training team members on ArgoCD fundamentals

Implementation Steps:

  1. Prepare your Git repository with Kubernetes manifests (Deployment and Service)
  2. Create the ArgoCD Application manifest pointing to your Git repository
  3. Apply the Application manifest to the ArgoCD namespace
  4. Verify the application is created and check its synchronization status
  5. Monitor the deployment to ensure pods are running and healthy

Validation Steps:

  • Check ArgoCD UI: Application should show "Healthy" and "Synced" status
  • Run `kubectl get pods -n nginx-demo` to verify nginx pods are running
  • Run `kubectl get svc -n nginx-demo` to verify the service is created
  • Test application accessibility by port-forwarding or using LoadBalancer
  • Verify auto-sync works by making a change in Git and watching it auto-deploy

Expected Outcomes:

  • 3 nginx replicas running in the nginx-demo namespace
  • Service exposing nginx on port 80
  • ArgoCD automatically syncing changes from Git to the cluster
  • Application health being monitored by ArgoCD

Application Manifest

apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: nginx-demo
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/your-org/argocd-demo.git
    targetRevision: main
    path: nginx-demo
  destination:
    server: https://kubernetes.default.svc
    namespace: nginx-demo
  syncPolicy:
    automated:
      prune: true
      selfHeal: true
    syncOptions:
    - CreateNamespace=true

Kubernetes Deployment (in Git repo)

apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment
  namespace: nginx-demo
spec:
  replicas: 3
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.21
        ports:
        - containerPort: 80
---
apiVersion: v1
kind: Service
metadata:
  name: nginx-service
  namespace: nginx-demo
spec:
  selector:
    app: nginx
  ports:
  - port: 80
    targetPort: 80
  type: LoadBalancer

Deployment Commands

argocd app create nginx-demo --file nginx-app.yaml
argocd app get nginx-demo
argocd app sync nginx-demo
argocd app list
Tip: Always start with manual sync operations to understand the behavior before enabling automated sync.

2. Complete GitOps Workflow Setup

Use Case

Overview: Set up a complete GitOps workflow with repository structure and deployment automation. This scenario demonstrates how to structure your Git repository for GitOps and implement proper separation of concerns using ArgoCD Projects.

What This Use Case Covers:

  • Designing a proper GitOps repository structure
  • Creating ArgoCD Projects for application isolation and governance
  • Setting up environment-specific application configurations
  • Implementing proper access control with project-level RBAC
  • Establishing deployment automation across environments

When to Use This Scenario:

  • You're setting up a new GitOps workflow from scratch
  • You need to organize multiple applications and environments
  • You want to implement proper access control and governance
  • You're establishing standards for team collaboration in GitOps
  • You need to separate deployment configurations from application code

Implementation Steps:

  1. Design and create the repository structure with separate directories for apps, charts, and projects
  2. Create an ArgoCD Project to define boundaries and access controls
  3. Create Application manifests for each environment (staging, production)
  4. Configure environment-specific values and configurations
  5. Set up proper RBAC policies within the project
  6. Apply the project configuration first, then the applications

Validation Steps:

  • Verify project creation: `argocd proj get guestbook-project`
  • Check that applications are grouped under the correct project
  • Test access control by trying to create applications outside project boundaries
  • Verify environment-specific configurations are applied correctly
  • Test the complete workflow from Git commit to deployment

Expected Outcomes:

  • Well-organized Git repository supporting multiple environments
  • ArgoCD Project with proper access controls and boundaries
  • Separate application deployments for staging and production
  • Team members can only deploy to approved namespaces and repositories
  • Clear separation between application code and deployment configuration

3. Multi-Environment Management

Use Case

Overview: Manage the same application across multiple environments (dev, staging, production) with different configurations. This scenario demonstrates how to use Kustomize to manage environment-specific variations while maintaining a single source of truth.

What This Use Case Covers:

  • Using Kustomize for environment-specific configuration management
  • Creating base configurations and environment overlays
  • Managing different replica counts, resources, and images per environment
  • Implementing proper namespace isolation between environments
  • Using ArgoCD Applications for each environment

When to Use This Scenario:

  • You need to manage the same application across multiple environments
  • You want to maintain DRY (Don't Repeat Yourself) principles
  • You need environment-specific configurations (different resources, replicas, etc.)
  • You're implementing a promotion workflow from dev to staging to production
  • You want to reduce configuration drift between environments

Implementation Steps:

  1. Create a base directory with common Kubernetes manifests
  2. Create environment-specific overlay directories (dev, staging, production)
  3. Define Kustomization files for each environment with specific patches
  4. Create ArgoCD Applications for each environment pointing to respective overlays
  5. Configure environment-specific parameters (replicas, resources, image tags)
  6. Apply applications and verify environment-specific configurations

Validation Steps:

  • Verify base configuration: `kubectl kustomize environments/base`
  • Check each environment's generated manifests: `kubectl kustomize environments/production`
  • Verify different replica counts in each environment
  • Check that resources are correctly applied per environment
  • Test that changes in base propagate to all environments
  • Verify environment-specific changes don't affect other environments

Expected Outcomes:

  • Single base configuration shared across all environments
  • Environment-specific overrides for replicas, resources, and images
  • Proper namespace isolation (dev, staging, production)
  • Easy promotion workflow between environments
  • Reduced configuration duplication and maintenance overhead

Helm Chart Application

apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: redis-ha
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/argoproj/argo-helm
    targetRevision: 3.5.0

  syncPolicy:
    automated:
      prune: true
      selfHeal: true
    syncOptions:
    - CreateNamespace=true

Kustomize Application

apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: myapp-kustomize
  namespace: argocd

  project: default
  source:
    repoURL: https://github.com/your-org/argocd-demo.git

- service.yaml

  app: myapp

namespace: production
namePrefix: prod-
bases:
- ../../base
patchesStrategicMerge:
- deployment-patch.yaml
- service-patch.yaml
configMapGenerator:
- name: myapp-config
  behavior: merge
  files:
  - config.properties
images:
- name: myapp
  newName: myapp
  newTag: v2.0.0

# kustomize/overlays/production/deployment-patch.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp

  replicas: 5
  template:
    spec:
      containers:
      - name: myapp
        resources:
          requests:
            memory: "512Mi"
            cpu: "500m"
          limits:
            memory: "1Gi"
            cpu: "1000m"

Manual Sync with Validation

apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: critical-app
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/your-org/argocd-demo.git
    targetRevision: main

What is GitOps?

GitOps is a modern approach to continuous delivery that uses Git as the single source of truth for infrastructure and application configurations. Instead of running commands manually, you make changes in Git, and those changes are automatically applied to your Kubernetes cluster.

Key Concepts Explained

GitOps Core Concepts:
1. Single Source of Truth: Your Git repository contains all configuration files
2. Desired State: What you define in Git is what should be running in the cluster
3. Synchronization: ArgoCD ensures the cluster matches the Git repository
4. Declarative Configuration: You describe WHAT you want, not HOW to achieve it

Traditional Deployment vs GitOps:
Traditional: 
  Developer writes code → CI builds → Manual kubectl commands → Cluster

GitOps:
  Developer writes code + manifests → CI builds → Git commit → ArgoCD auto-syncs → Cluster

Basic ArgoCD Architecture

ArgoCD Components:
┌─────────────────────────────────────────────────────────────┐
│                     Git Repository                          │
│  (Contains all Kubernetes manifests and configurations)    │
└──────────────────────────┬──────────────────────────────────┘
                           │
                           │ ArgoCD monitors
                           │
┌──────────────────────────▼──────────────────────────────────┐
│                    ArgoCD Controller                         │
│  - Watches Git repository for changes                        │
│  - Compares desired state (Git) vs actual state (cluster)    │
│  - Synchronizes differences automatically                   │
└──────────────────────────┬──────────────────────────────────┘
                           │
                           │ Applies changes
                           │
┌──────────────────────────▼──────────────────────────────────┐
│                  Kubernetes Cluster                          │
│  (Runs the actual applications and services)                 │
└─────────────────────────────────────────────────────────────┘

Basic Auto Sync Configuration

# File: auto-sync-app.yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: auto-sync-demo

spec:

Auto Sync Options Explained

syncPolicy.automated.prune:
  - When true: Resources deleted from Git are deleted from cluster
  - When false: Resources deleted from Git remain in cluster
  - Use case: Prevent accidental deletion of important resources

syncPolicy.automated.selfHeal:
  - When true: Manual changes to cluster are automatically reverted
  - When false: Manual changes are preserved (creates drift)
  - Use case: Allow temporary manual changes

Example scenarios:

Prune: true, SelfHeal: true
  - Full automation
  - Cluster always matches Git exactly
  - Good for stateless applications

Prune: false, SelfHeal: true
  - Resources never auto-deleted
  - Manual changes are reverted
  - Good for databases with persistent data

Prune: true, SelfHeal: false
  - Resources auto-deleted
  - Manual changes are preserved
  - Good for emergency manual fixes

4. No Complex Commands: No need for complex kubectl rollback commands

  (Simple, uses familiar Git commands)

Rollback Scenario

Scenario: You deployed version 2.0 but it has bugs, need to rollback to 1.0

Current state:
- Git main branch: version 2.0 (broken)
- Cluster: running version 2.0 (broken)
- Need to rollback to: version 1.0 (working)

Rollback Method 1: Git Revert

# Step 1: Identify the problematic commit
git log --oneline

# Output:
# a1b2c3d Update to version 2.0 (broken)
# e5f6g7h Update to version 1.5
# h8i9j0k Initial version 1.0

# Step 2: Revert the problematic commit
git revert a1b2c3d

# This creates a new commit that undoes the changes
# The commit message will be something like:
# "Revert 'Update to version 2.0'"

# Step 3: Push the revert
git push origin main

# Step 4: ArgoCD automatically syncs (if auto-sync enabled)
# Or manually sync if using manual sync
argocd app sync myapp

# Step 5: Verify the rollback
kubectl get pods -n myapp
# Should see version 1.0 running again

Rollback Method 2: Git Reset

# Use this method if you want to completely remove the bad commits

# Step 1: Reset to the working commit
git reset --hard h8i9j0k  # Reset to version 1.0 commit

# Step 2: Force push (use with caution!)
git push --force origin main

# Step 3: ArgoCD will detect the change and sync
argocd app sync myapp

# Warning: This rewrites Git history and can affect other team members!
# Use git revert instead for collaborative environments.

Rollback Method 3: Using ArgoCD Revision

# Step 1: Find the working revision
argocd app history myapp

# Output:
# ID    DATE             REVISION   STATUS
# 1     2024-01-15       h8i9j0k    Synced
# 2     2024-01-16       e5f6g7h    Synced
# 3     2024-01-17       a1b2c3d    Synced (current, broken)

# Step 2: Sync to the working revision
argocd app sync myapp --revision h8i9j0k

# Step 3: Verify the rollback
argocd app get myapp
kubectl get pods -n myapp

Rollback Method 4: Using Git Checkout

# Create a rollback branch and switch to it
git checkout -b rollback-to-v1 h8i9j0k

# Switch main branch to point to the rollback
git checkout main
git reset --hard rollback-to-v1

# Push the rollback
git push --force origin main

# Clean up
git branch -D rollback-to-v1

# ArgoCD will sync the rollback
argocd app sync myapp

Understanding Kubernetes Probes

Kubernetes provides three types of probes:

1. Liveness Probe
   - Checks if the container is still running
   - If failed, Kubernetes restarts the container
   - Prevents dead containers from running

2. Readiness Probe
   - Checks if the container is ready to serve traffic
   - If failed, removes pod from service endpoints
   - Prevents traffic from going to unready pods

3. Startup Probe
   - Checks if the application has started
   - Disables other probes until startup succeeds
   - Useful for slow-starting applications

Why health checks matter:
- Automatic recovery from failures
- Prevents routing to unhealthy instances
- Ensures high availability
- Provides monitoring insights

HTTP Health Checks

# File: deployment-with-http-probes.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp-with-probes
  namespace: myapp
spec:
  replicas: 3
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
      - name: myapp
        image: myapp:latest
        ports:
        - containerPort: 8080
        # Liveness probe - checks if app is alive
        livenessProbe:
          httpGet:
            path: /health
            port: 8080
          initialDelaySeconds: 30  # Wait 30s before first check
          periodSeconds: 10        # Check every 10s
          timeoutSeconds: 5         # Timeout after 5s
          failureThreshold: 3      # Fail after 3 consecutive failures
        # Readiness probe - checks if app is ready
        readinessProbe:
          httpGet:
            path: /ready
            port: 8080
          initialDelaySeconds: 10
          periodSeconds: 5
          timeoutSeconds: 3
          failureThreshold: 2
        # Startup probe - for slow-starting apps
        startupProbe:
          httpGet:
            path: /startup
            port: 8080
          initialDelaySeconds: 0
          periodSeconds: 5
          timeoutSeconds: 3
          failureThreshold: 30  # Gives 150s total (30 * 5s)

# /health endpoint (liveness probe)

    # Check if critical services are running

# /ready endpoint (readiness probe)
@app.route('/ready')
def ready():
    # Check if app can handle requests
    if not cache.is_warm():
        return Response(status=503)
    if not dependencies.are_loaded():
        return Response(status=503)
    return Response(status=200)

@app.route('/startup')
def startup():
    # Check if application has fully started
    if not app.is_initialized():
        return Response(status=503)
    return Response(status=200)
