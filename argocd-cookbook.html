A. Title and Overview

Argo CD GitOps Operator

Role in Project: Argo CD serves as the GitOps operator that automatically synchronizes Kubernetes manifests from Git to the cluster, ensuring the deployed state matches the desired state defined in the repository.

Why Argo CD is Used

  • GitOps Native: Implements GitOps methodology with Git as single source of truth
  • Automated Sync: Automatically detects and applies Git changes to Kubernetes
  • Self-Healing: Automatically reverts manual changes to maintain desired state
  • Rollback Support: Easy rollback to previous application states
  • Declarative Configuration: Infrastructure and deployment as code with YAML manifests

GitOps Pipeline Position

Argo CD is the CD (Continuous Deployment) component of the GitOps pipeline. It monitors the Git repository and automatically applies Kubernetes manifests to the cluster, ensuring that the deployed state always matches the Git repository state.

Key Responsibilities

  • Monitor Git repository for manifest changes
  • Automatically sync changes to Kubernetes cluster
  • Detect and alert on configuration drift
  • Provide rollback capabilities for quick recovery
  • Manage application lifecycle and health monitoring
  • Implement RBAC and access control

    D. Installation / Setup

    Step 1: Create Argo CD Namespace

    # Create namespace for Argo CD
kubectl create namespace argocd

    Step 2: Install Argo CD

    # Install ArgoCD using official manifests
kubectl apply -n argocd -f https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/install.yaml

    Step 3: Apply Custom Configuration

    # Apply custom ConfigMaps
kubectl apply -f argocd/argocd-cm.yaml -n argocd
kubectl apply -f argocd/argocd-rbac-cm.yaml -n argocd

    Step 4: Create Project

    # Create ArgoCD project
kubectl apply -f argocd/project.yaml -n argocd

    Step 5: Create Application

    # Create ArgoCD application
kubectl apply -f argocd/application.yaml -n argocd

    Step 6: Validate Installation

    # Check ArgoCD pods
kubectl get pods -n argocd

# Check ArgoCD application
kubectl get application -n argocd

# Access ArgoCD UI (port-forward)
kubectl port-forward svc/argocd-server -n argocd 8080:443

# Get initial admin password
kubectl -n argocd get secret argocd-initial-admin-secret -o jsonpath='{.data.password}' | base64 -d
    Tip: For production deployments, consider using Helm to install Argo CD for easier configuration and management.

    Common Configuration Mistakes

    Warning: Avoid enabling automated prune without understanding the consequences. It will remove resources from Kubernetes that are not present in Git, which can cause data loss if not configured properly.
    • Enabling automated prune without proper testing
    • Incorrect repository URL or branch configuration
    • Wrong namespace configuration causing deployment failures
    • Missing resource whitelisting causing sync failures
    • Incorrect sync policy causing deployment issues

    F. Cookbook Recipes

    Recipe 1: Create Basic ArgoCD Application

    Purpose: Set up basic ArgoCD application for GitOps deployment

    Ingredients: Argo CD installed, Git repository, Kubernetes manifests

    Steps:

    1. Create ArgoCD project with repository configuration
    2. Configure allowed destinations and resource whitelists
    3. Create Application YAML manifest
    4. Configure source repository and target cluster
    5. Set sync policy to manual initially
    6. Apply manifest: kubectl apply -f application.yaml
    7. Test manual sync with ArgoCD CLI

    Expected Result: ArgoCD application ready for GitOps deployment

    Verification: argocd app get application-name to check status

    Recipe 2: Configure Automated Sync with Self-Healing

    Purpose: Enable automated GitOps workflow with self-healing capabilities

    Ingredients: ArgoCD CLI, application manifest

    Steps:

    1. Update application sync policy to enable automated mode
    2. Set automated.prune: true for cleanup
    3. Set automated.selfHeal: true for self-healing
    4. Add syncOptions with CreateNamespace=true
    5. Apply updated manifest
    6. Test by making changes in Git repository

    Expected Result: Changes in Git automatically sync to cluster

    Verification: Monitor application status after Git changes

    Recipe 3: Configure Multi-Environment Deployment

    Purpose: Set up ArgoCD for managing multiple environments

    Ingredients: Multiple application manifests, project configuration

    Steps:

    1. Create project with multiple destination namespaces
    2. Create separate applications for each environment
    3. Configure different target branches for each environment
    4. Set environment-specific sync policies
    5. Test deployment to each environment

    Expected Result: Multi-environment GitOps deployment

    Verification: Check each application's status and sync status

    Recipe 4: Configure Rollback with ArgoCD

    Purpose: Enable quick rollback to previous application states

    Ingredients: ArgoCD CLI, application history

    Steps:

    1. View application history: argocd app history application-name
    2. Rollback to previous revision: argocd app rollback application-name
    3. Rollback to specific revision: argocd app rollback application-name --revision 2
    4. Verify rollback completed successfully
    5. Test application functionality

    Expected Result: Application reverted to previous state

    Verification: Check application status and pod versions

    Recipe 5: Configure Health Checks and Monitoring

    Purpose: Enable application health monitoring and alerting

    Ingredients: ArgoCD application manifest, health endpoints

    Steps:

    1. Configure health checks in application manifest
    2. Set up appropriate health check endpoints
    3. Configure ArgoCD to monitor application health
    4. Set up notifications for health status changes
    5. Test health check functionality

    Expected Result: Application health monitored and reported

    Verification: Check application health status in ArgoCD UI

    • Output: Deployed resources in cluster, sync status, health status

    System Integration

    • Git Repository: Source of truth for manifests
    • Kubernetes Cluster: Deployment target for ArgoCD
    • GitHub Actions: Triggers ArgoCD sync after deployment
    • Artifactory: Provides Docker images for deployment
    • Flask Application: Main application managed by ArgoCD

    GitOps Flow

    1. Developer commits Kubernetes manifest changes to Git repository
    2. ArgoCD detects changes in Git repository
    3. ArgoCD compares desired state (Git) with actual state (cluster)
    4. ArgoCD syncs differences to bring cluster to desired state
    5. Kubernetes applies changes using rolling updates
    6. Application is deployed with new configuration

    J. Troubleshooting

    Problem: Application stuck in OutOfSync state ▼

    Symptoms: Application shows OutOfSync status and won't sync automatically

    Likely Cause: Invalid configuration, repository access issues, resource conflicts

    How to Diagnose: Check application details, review sync status, examine application logs

    Fix: Fix configuration, resolve repository issues, resolve conflicts

    Prevention: Test configuration before applying, validate repository access, use proper resource whitelisting

    Problem: Sync failures with permission errors ▼

    Symptoms: Sync fails with "resource not permitted" errors

    Likely Cause: Resource not in project whitelist, RBAC issues, namespace conflicts

    How to Diagnose: Check project configuration, review RBAC policies, examine sync logs

    Fix: Update project whitelisting, fix RBAC policies, resolve namespace conflicts

    Prevention: Configure proper resource whitelisting, test RBAC policies, use namespace isolation

    Problem: Health checks failing ▼

    Symptoms: Application marked unhealthy, sync issues

    Likely Cause: Incorrect health endpoint, application not ready, network issues

    How to Diagnose: Check application health endpoints, review pod status, test network connectivity

    Fix: Fix health endpoints, wait for application readiness, resolve network issues

    Prevention: Test health endpoints, configure appropriate timeouts, implement proper startup probes

    K. Q&A / FAQ

    Q: Why is Argo CD used instead of kubectl apply?
    A: Argo CD provides automated sync, self-healing, health monitoring, and rollback capabilities that kubectl apply doesn't have. It implements the GitOps methodology with Git as the single source of truth.
    Q: What happens if ArgoCD configuration is wrong?
    A> Sync operations will fail or produce incorrect deployments. Always validate configuration with dry-run and test in non-production environments before enabling automated sync.
    Q: How do I validate ArgoCD configuration?
    A> Use argocd app get application-name to check status, argocd app diff application-name to see differences, and test with manual sync before enabling automation.
    Q: How do I debug ArgoCD sync failures?
    A> Check application details in ArgoCD UI, review sync status logs, examine application controller logs, and use argocd CLI for detailed diagnostics.
    Q: How do I rollback ArgoCD deployment?
    A> Use argocd app rollback application-name to rollback to previous version, or argocd app rollback application-name --revision 2 for specific revision.
    Q: What logs should I inspect for troubleshooting?
    A> ArgoCD application logs, controller logs, repository server logs, application controller logs, and Kubernetes pod logs for the application.
    Q: How do I secure ArgoCD credentials?
    A> Use RBAC for access control, use service accounts for automation, integrate with secret management systems, and regularly rotate credentials.
    Q: How does ArgoCD behave in production?
    A> In production, ArgoCD handles high-traffic sync operations, provides enterprise-grade reliability, integrates with monitoring systems, and supports multi-cluster and multi-region deployments.
    Q: How do I improve ArgoCD performance?
    A> Optimize repository access, use efficient diff algorithms, configure appropriate sync intervals, optimize application controller settings, and tune resource limits.
